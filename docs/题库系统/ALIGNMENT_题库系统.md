## 2. 业务需求分析

### 2.1 核心业务场景

#### 场景1：督学布置刷题任务
```
督学人员 → 在督学系统创建作业任务
         → 指定模块（如"图形推理50题"）
         → 学员进入题库系统刷题
         → 题库系统记录答题情况
         → 同步薄弱项到督学系统
```

#### 场景2：学员自主刷题
```
学员 → 登录题库系统
     → 选择模块进行练习
     → 完成后查看解析
     → 系统自动记录错题
     → 生成薄弱项报告
```

#### 场景3：套卷模拟考试
```
学员 → 选择套卷（如"2025国考行测真题"）
     → 限时作答（120分钟）
     → 交卷后查看成绩和解析
     → 系统分析各模块得分情况
```

#### 场景4：错题回顾
```
学员 → 进入错题本
     → 按模块/时间筛选错题
     → 重新作答
     → 查看进步情况
```

### 2.2 核心痛点

| 痛点 | 具体表现 | 影响程度 |
|------|---------|---------|
| 纸质刷题难追踪 | 学员用纸质做题，数据无法沉淀 | ⭐⭐⭐⭐⭐ |
| 正确率靠人工统计 | 督学人员手动录入成绩 | ⭐⭐⭐⭐ |
| 薄弱项难以精准识别 | 无法细化到具体知识点 | ⭐⭐⭐⭐ |
| 错题复习不系统 | 学员无法追踪错题改进 | ⭐⭐⭐ |
| 刷题资料分散 | 各种PDF、Word文档难以管理 | ⭐⭐⭐ |

## 3. 系统目标

### 3.1 业务目标

1. **刷题数字化**：学员所有刷题记录100%线上化
2. **薄弱项精准识别**：细化到三级知识点
3. **错题系统化**：自动收集错题，支持回顾练习
4. **与督学系统联动**：实时同步学习数据

### 3.2 技术目标

1. 支持30-100名学员同时在线刷题
2. 页面加载时间 < 2秒
3. 题库容量支持10000+题目
4. 与督学系统API无缝对接

### 3.3 可量化指标

- 学员刷题数据记录完整率：100%
- 薄弱项识别准确率：>80%
- 错题回顾覆盖率：100%
- 系统可用性：>99%

## 5. 技术约束

### 5.1 技术栈要求

为与督学系统保持一致，采用相同技术栈：

| 层次 | 技术选型 | 说明 |
|------|---------|------|
| 后端框架 | Python Flask | 与督学系统一致 |
| ORM | SQLAlchemy | 与督学系统一致 |
| 数据库 | SQLite/PostgreSQL | 开发用SQLite |
| 前端框架 | Bootstrap 5 | 与督学系统一致 |
| 认证方式 | Flask-Login | 与督学系统一致 |

### 5.2 部署约束

- **开发环境**：Mac
- **生产环境**：与督学系统同服务器部署
- **域名**：可使用子域名（如 tiku.xxx.com）

### 5.3 数据约束

- 题目数量：初期1000-5000题，远期10000+
- 学员数量：30-100人
- 单次刷题：最多200题/次
